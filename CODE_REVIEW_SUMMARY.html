⚠️ CRITICAL ARCHITECTURAL FINDING

There is a fundamental mismatch between your specification and the existing system architecture.

Your Spec Says: Read CSV → Apply rules → Write to CSV (standalone, no database)

Existing System Is: Database-driven, multi-tenant, all operations require AsyncSession

The Core Problem

Email Sending Requires Database

async def send_email(
    to: str,

    db: AsyncSession,  # ← REQUIRED!
    ...
) -> dict:
    limit = _effective_limit(smtp)  # fetches from db

        return {"error": "Daily limit reached"}
    # ... send ...

Problem: Cannot send email without checking daily limits from database

WhatsApp Sending Requires Database + Client ID

async def send_whatsapp(
    to: str,
    body: str,
    client_id: uuid.UUID,  # ← REQUIRED! (multi-tenant)
    db: AsyncSession,      # ← REQUIRED!
    ...
) -> dict:
    settings = await get_settings(db)  # fetch credentials
    phone_number = await get_number_for_client(client_id, db)
    # ... send via Meta API ...
    phone_number.sent_today += 1  # update db

Problems:

  • Requires client_id to route to correct WhatsApp phone number
  • Requires database to fetch encrypted WhatsApp credentials
  • Requires database to track daily limits per phone number

Three Implementation Paths

Option A: CSV-Only (Matches Your Spec Literally)

Read/write CSV directly with no database involvement.

✓ Pros:
  • Matches your specification exactly
  • Simplest to understand and debug
  • Completely standalone, no multi-tenant complexity
  • Can run without PostgreSQL/Redis if needed
✗ Cons:
  • Cannot reuse existing send_email() or send_whatsapp() functions
  • Must create new lightweight sending code OR refactor existing functions
  • No daily limit tracking (would need CSV headers or separate config)
  • WhatsApp sends without credentials validation (could fail at runtime)
  • Email sends without SMTP account selection (which account to use?)
  • Isolated from multi-tenant system — can't scale to multiple clients

Effort: Medium (need to write new send code)

Risk: High (untested send paths, no reuse of proven code)

Option B: Light Database Integration (RECOMMENDED)

Leads live in database, but use lightweight tracking (not the complex Campaign system).

✓ Pros:
  • Reuses proven email_service and whatsapp_service code
  • Reuses existing credential management and daily limit logic
  • Can still import from CSV (existing csv_import_service)
  • Aligns with multi-tenant architecture (ready to scale)
  • Cleaner separation of concerns
  • Can generate reports from database later
✗ Cons:
  • Requires learning database model structure
  • Need to set up SMTP account and WhatsApp credentials in database
  • Need to create a Client record (or use existing one)
  • Slightly more code than Option C, but much less

Effort: Low-Medium (use existing services, add tracking table)

Risk: Low (reuses tested code)

Option C: Full Campaign System

Import CSV → create Campaign with 4 Steps → use existing campaign execution.

✓ Pros:
  • Zero new code needed (mostly configuration)
  • Reuses all existing monitoring, retry logic, and reporting
  • Most robust and feature-complete
  • Can manage campaigns from UI
✗ Cons:
  • Overkill for a simple CSV-based scheduler
  • Adds unnecessary complexity (pacing, enrollments, etc.)
  • Requires understanding campaign/enrollment model

Effort: Very Low (mostly API calls)

Risk: Very Low (proven system)
